流式布局、首页、搜索与好友入口

一、目标

本节主要完成展示层与信息流相关功能：

  • 个人空间页面；
  • 角色卡片组件；
  • 无限滚动 / 流式加载；
  • 首页角色广场；
  • 搜索功能；
  • 好友模型与好友列表；
  • 点击角色进入聊天弹窗的前置逻辑。

涉及的重要知识点：

  • IntersectionObserver；
  • CSS Grid 自适应布局；
  • Django 分页查询；
  • Django ORM Q 对象；
  • 前端 query 参数搜索；
  • 事件冒泡与 @click.stop；
  • 外键级联删除。

十四、易错点总结

  1. 首页接口返回字段应统一为 result，不要写成 results；
  2. items_count 从 query 参数取出是字符串，要转成 int；
  3. 前端接口路径建议以 / 开头；
  4. IntersectionObserver 创建后必须 observe(sentinelRef.value)；
  5. 组件卸载时必须 disconnect()；
  6. 删除按钮要用 @click.stop；
  7. 搜索 query 改变后要清空旧列表再重新请求；
  8. 个人空间从一个用户切换到另一个用户时，要监听 route.params.user_id 并重置页面；
  9. 当前项目接口 get_or_creat 拼写特殊，调用时必须与后端一致。

十五、小结

本节把项目从“单点功能”扩展成“可浏览内容的平台”：

  • 个人空间展示用户与角色；
  • 首页展示全站角色；
  • 搜索通过 URL query 和后端 Q 查询实现；
  • 无限滚动通过哨兵与 IntersectionObserver 实现；
  • 角色卡片成为后续聊天入口；
  • Friend 模型为每个用户和角色之间建立独立聊天关系。

十六、新手版：什么是信息流和无限滚动

首页角色广场、个人空间角色列表都属于“信息流”。

你可以把信息流想象成短视频 App 或商品列表：

  • 一开始不加载全部内容；
  • 先加载一部分；
  • 用户滚动到底部时再加载下一部分；
  • 这样页面更快，也不会一次请求太多数据。

16.1 为什么不能一次加载全部角色

如果数据库里只有 10 个角色，一次加载没问题。

但如果将来有 10 万个角色，一次性返回会导致：

  • 后端查询慢；
  • 网络传输慢；
  • 浏览器渲染卡顿；
  • 用户等待时间长。

所以本项目每次只取 20 条。

16.2 哨兵是什么

哨兵就是列表底部的一个小元素。

它本身不重要，重要的是：当浏览器看到它时，说明用户快滚到底了，可以加载更多。

十八、跟做步骤：首页搜索

18.1 前端导航栏

  1. 搜索框用 v-model 绑定 searchQuery；
  2. 表单提交时调用 handleSearch；
  3. 使用 router.push 跳转首页；
  4. 把搜索词放进 URL query。

18.2 首页页面

  1. 从 route.query.q 读取搜索词；
  2. 请求后端时传 search_query；
  3. 监听 query 改变；
  4. query 改变后清空列表并重新加载。

18.3 后端接口

  1. 从 request.query_params 读取 search_query；
  2. 如果为空，查询全部；
  3. 如果不为空，用 Q 查询名字或简介；
  4. 分页返回。

二十、新手重点：事件冒泡

假设角色卡片是一个大按钮，点击卡片会打开聊天框。

卡片里面还有一个删除按钮。如果点击删除按钮时不阻止冒泡，浏览器会认为你同时点击了：

  1. 删除按钮；
  2. 外层角色卡片。

结果就是：删除的同时还打开了聊天框。

解决：

.stop 表示事件到这个按钮就停止，不再传给外层。

二十一、常见报错排查

21.1 页面一直显示加载中

检查：

  1. isLoading 是否在 finally 中恢复为 false；
  2. 接口是否报错；
  3. data.result 是否和后端返回字段一致；
  4. 后端是否误写成 results；
  5. Network 中请求是否成功。

21.2 无限重复请求

检查：

  1. hasCharacters 是否在无数据时设为 false；
  2. isLoading 是否能阻止重复请求；
  3. checkSentinelVisible() 是否逻辑错误；
  4. observer.observe 是否监听了正确元素。

21.3 搜索后结果没变

检查：

  1. 导航栏是否把搜索词写进 URL；
  2. 首页是否监听 route.query.q；
  3. 请求是否传了 search_query；
  4. 后端是否使用 Q(name__icontains=...)；
  5. 搜索后是否清空旧列表。

二十七、小练习与自测答案

练习 1

问题：为什么无限滚动需要 isLoading？

答案：因为哨兵可能连续多次进入视口，如果没有 isLoading 防抖，会同时发出多个重复请求，造成列表重复和后端压力。

练习 2

问题：搜索后为什么要把旧列表清空？

答案：搜索条件变了，旧列表已经不属于新搜索结果。如果不清空，新旧结果会混在一起。

练习 3

问题：为什么点击角色卡片前要先获取或创建 Friend？

答案：聊天记录和记忆是按 Friend 保存的。只有拿到 friend_id，聊天模块才知道当前用户是在和哪个角色建立的哪一段关系聊天。
