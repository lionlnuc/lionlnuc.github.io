文字聊天、SSE、记忆、工具调用与 RAG

一、目标

本节实现 AI 角色的文字聊天能力，并逐步扩展为具备上下文、记忆、工具调用和知识库检索的 Agent。

核心内容包括：

  • 消息数据库 Message；
  • LangGraph 对话流程；
  • SSE 流式输出；
  • 前端流式请求封装；
  • 聊天历史加载；
  • 系统提示词；
  • 短期记忆；
  • 长期记忆；
  • Function Calling / Tool Calling；
  • LanceDB + Embedding + RAG。

二、聊天数据模型

2.1 Friend 与 Message 的关系

含义：

  • 一个用户可以和多个角色建立好友关系；
  • 一个角色也可以被多个用户聊天；
  • 每个 Friend 保存某个用户与某个角色之间的独立聊天记录；
  • Message 归属于某个 Friend。

2.2 Message 模型

字段说明：

字段 说明
friend 当前对话属于哪段用户-角色关系
user_message 用户原始输入
input 实际传给大模型的上下文，包括系统提示词、历史消息等
output AI 回复
input_tokens 输入 token 数
output_tokens 输出 token 数
total_tokens 总 token 数

2.3 为什么要保存 input？

保存 input 便于：

  • 调试模型为什么这样回答；
  • 分析提示词是否正确；
  • 排查上下文记忆是否拼接错误；
  • 做 token 成本统计。

十二、SSE 与 WebSocket 对比

对比项 SSE WebSocket
通信方向 服务器到客户端单向 双向
协议 HTTP 独立协议升级
自动重连 浏览器原生支持 需要自己实现
复杂度 低 较高
适合场景 大模型流式文本、通知 即时通讯、实时游戏

本项目文字流式输出使用 SSE 更合适，因为 AI 回复主要是服务端向客户端推送。

十三、易错点总结

  1. SSE 每条消息要以 data: 开头，并以 \n\n 结束；
  2. 流结束要发送 data: [DONE]\n\n；
  3. StreamingHttpResponse 要设置 content_type='text/event-stream'；
  4. Nginx 部署时要关闭缓冲：X-Accel-Buffering: no；
  5. 前端解析 [DONE] 时不要再 JSON.parse；
  6. handlePushFrontMessage 用 unshift，不是 shift；
  7. 加载历史时要用 last_message_id，避免和前端临时消息错位；
  8. 聊天气泡头像要用 :src，不要写成字符串 src="character.photo"；
  9. 工具函数必须写清楚 docstring；
  10. .env 和知识库数据不应提交到 Git。

十四、小结

本节实现了 AgentTalk 的核心 AI 聊天能力：

  • 后端用 LangGraph 封装对话流程；
  • 使用 SSE 实现大模型流式输出；
  • 前端使用 fetch-event-source 接收流；
  • 消息保存到 Message 表；
  • 历史记录支持向上滚动加载；
  • 系统提示词、短期历史、长期记忆共同构成上下文；
  • Tool Calling 让模型能调用外部函数；
  • RAG 让模型能查阅项目私有知识库。

二十二、常见报错排查

22.1 前端收不到流式数据

检查：

  1. 后端是否使用 StreamingHttpResponse；
  2. 响应头是否是 text/event-stream；
  3. 每条数据是否以 data: 开头；
  4. 每条数据是否以 \n\n 结尾；
  5. 前端是否使用 fetchEventSource；
  6. Network 中 Response 是否持续变化。

22.2 前端 JSON.parse 报错

常见原因：收到 [DONE] 后还尝试 JSON.parse('[DONE]')。

正确逻辑：

22.3 聊天记录顺序反了

后端通常按 -id 取最近 10 条，取出后需要反转成从旧到新。

22.4 AI 回复没有上下文

检查：

  1. 是否调用了 add_system_prompt；
  2. 是否调用了 add_recent_message；
  3. Message 表是否真的有历史记录；
  4. 拼接后的 inputs['messages'] 是否正确。

22.5 工具调用不生效

检查：

  1. 工具函数是否加了 @tool；
  2. 工具函数是否有清晰 docstring；
  3. 是否执行了 .bind_tools(tools)；
  4. LangGraph 是否加了 ToolNode；
  5. 条件边是否能进入 tools 节点。

二十五、关键代码逐行解释示例

25.1 后端 SSE 返回

逐行解释：

代码 新手解释
def event_stream(): 定义一个生成器函数，用来一点一点返回数据
for chunk in graph.stream(inputs): 从大模型流程中不断取出新生成的片段
content = chunk.get(...) 取出当前片段中的文本内容
if content: 空内容不发送，避免前端收到无意义数据
yield f"data: ...\n\n" 按 SSE 格式返回一段数据
yield "data: [DONE]\n\n" 告诉前端本次流式输出结束

25.2 前端接收 SSE

逐行解释：

代码 新手解释
fetchEventSource(url, {...}) 发起一个能接收 SSE 的请求
method: 'POST' 聊天消息通过 POST 提交
headers 携带 Token 和 Content-Type
body 把 friend_id 和 message 转成 JSON 字符串
onmessage(event) 每收到一段 SSE 数据就执行一次
event.data === '[DONE]' 如果后端说结束，就不再解析 JSON
JSON.parse(event.data) 把后端返回的字符串转成对象
+= payload.content 把新片段追加到最后一条 AI 消息

二十八、小练习与自测答案

练习 1

问题：为什么流式聊天的响应不是普通 JSON？

答案：普通 JSON 需要等全部内容生成完才能返回。SSE 可以边生成边返回，让用户更快看到回复。

练习 2

问题：为什么前端把 AI 消息放在 history 最后一项？

答案：后端流式返回的是一段一段文本。前端每收到一段，都追加到最后一条 AI 消息中，最终拼成完整回复。

练习 3

问题：为什么后端查询 Friend 时必须限制当前用户？

答案：防止用户传入别人的 friend_id 读取或写入别人的聊天记录。这是权限安全检查。
