  • 图片裁剪 Croppie；
  • Base64 转 File；
  • Django ImageField；
  • 删除旧媒体文件；
  • Django ORM 外键查询；

九、易错点总结

  1. 默认头像路径要写对：user/photos/default.png；
  2. 删除旧头像时不能删除默认头像；
  3. 后端返回图片要使用 .url；
  4. Base64 转 File 前要确认是 data: 开头；
  5. get() 返回对象，filter() 返回 QuerySet；
  6. 修改角色时必须用 author__user=request.user 限制权限；
  7. 删除角色时先获取实例再删图片，最后再 delete()；
  8. Croppie 组件卸载时要 destroy()；
  9. 前端组件 props 异步更新时要用 watch 同步本地 ref。

十、小结

本节实现了用户资料编辑和角色 CRUD。核心是：

  • 前端用组件化方式管理复杂表单；
  • 图片使用 Croppie 裁剪并以 Base64 暂存；
  • 提交前转为 File 并放入 FormData；
  • 后端通过 request.FILES 接收文件；
  • 更新文件时删除旧文件，避免媒体目录膨胀；
  • 角色所有权通过 ORM 跨表查询保证安全。

十五、小练习与自测答案

练习 1

问题：为什么上传图片时不能直接用普通 JSON？

答案：JSON 适合传文本结构，不适合直接传二进制文件。图片文件通常使用 multipart/form-data，前端用 FormData 组织，后端从 request.FILES 读取。

练习 2

问题：删除角色时为什么查询条件要包含 author__user=request.user？

答案：这是权限校验。它保证用户只能删除自己创建的角色，不能通过猜测别人的 character_id 删除别人的角色。

练习 3

问题：为什么更新头像时要删除旧文件？

答案：如果不删除旧文件，媒体目录会不断堆积无用图片，占用磁盘空间。

十一、新手版：为什么图片上传比普通文本复杂

普通文本提交很简单：

但是图片不是普通字符串，而是二进制文件。浏览器不能直接把图片像普通 JSON 一样轻松传给后端，所以需要 FormData。

可以把 FormData 想象成一个快递箱：

  • 可以放文字纸条，例如用户名；
  • 也可以放真实文件，例如头像；
  • 后端收到后，从 request.data 拿文字，从 request.FILES 拿文件。

十五、常见报错排查

15.1 后端提示 photo 为空

检查：

  1. 前端是否使用了 FormData；
  2. 是否执行了 formData.append('photo', file)；
  3. 字段名是否和后端 request.FILES.get('photo') 一致；
  4. Base64 是否成功转成 File。

15.2 图片显示不出来

检查：

  1. 后端返回的是不是 .url；
  2. MEDIA_URL 和 MEDIA_ROOT 是否配置；
  3. 开发环境是否在根路由中加了 static('/media/', ...)；
  4. 前端如果在 Vite 开发环境运行，是否需要拼接后端 BASE_URL。
